This section describes the FeBEx architecture and the key design decisions that enable in-network duplicate suppression for federated LoRaWAN backhaul. Figure~\ref{fig:febexarchitecture} illustrates the overall system and the sequence of processing stages applied to each uplink packet inside the programmable switch. At a high level, FeBEx is deployed as a programmable aggregation switch between multiple gateways and shared backend services. Rather than allowing every received copy of an uplink packet to traverse the backhaul and be discarded only at the backend, FeBEx suppresses redundant copies directly in the forwarding path while preserving tenant-specific routing and attribution semantics.

\fakepar{System Overview}
FeBEx consists of two main components: a programmable data plane and a lightweight control plane. The data plane is implemented in P4 on BMv2 and is responsible for per-packet forwarding decisions. The control plane is implemented in Python using P4Runtime and Finsy, and is responsible for installing forwarding rules, configuring clone sessions, and rotating epochs for deduplication state expiry. Within the FeBEx pipeline, each packet is first parsed to extract the metadata fields, then matched to the appropriate tenant backend, checked against deduplication state, and finally cloned if it is the first valid copy of a logical uplink. In this way, FeBEx performs three tasks in a single forwarding pipeline: \textit{(i)} tenant-aware forwarding to the appropriate LoRa Network Server (LNS), \textit{(ii)} in-network duplicate suppression using compact register-based state, and \textit{(iii)} cloning of selected packets to a cloud endpoint for attribution.

\fakepar{Deployment Topology}
FeBEx assumes a star-shaped aggregation topology in which multiple gateways connect to a single programmable switch, which then forwards traffic to one of several tenant backends and, when required, to a cloud endpoint. In our Mininet-based implementation, gateway hosts generate packets and inject them into the switch, the BMv2 switch executes the FeBEx pipeline, tenant LNS hosts receive forwarded packets, and a separate cloud host receives cloned receipts. Concretely, if there are $K$ gateways and $M$ tenant backends, gateway hosts connect to ports $1 \ldots K$, tenant hosts connect to ports $K+1 \ldots K+M$, and the cloud host connects to port $K+M+1$. This topology captures the central aggregation role of FeBEx while remaining simple enough for controlled experimentation.

\fakepar{Packet Abstraction}
Rather than parsing the full LoRaWAN stack inside the switch, our prototype uses a compact custom metadata header that carries only the fields needed for deduplication, forwarding, and attribution, as shown in Fig.~\ref{fig:febexheader}. Each test packet consists of Ethernet, IPv4, UDP, a 12-byte FeBEx metadata header, and an arbitrary payload:
\begin{verbatim}
            Ethernet | IPv4 | UDP (port 5555) |  FeBEx Meta  | Payload
            
                                              |-- 12 bytes --|
\end{verbatim}
The metadata header contains four fields of interest: \texttt{dev\_addr}, \texttt{fcnt}, \texttt{gw\_id}, and two reserved bytes. The pair $(\texttt{dev\_addr}, \texttt{fcnt})$ identifies a logical uplink, while \texttt{gw\_id} records which gateway produced the current copy. This abstraction lets us focus on the core network problem---duplicate suppression in shared backhaul---without tying the design to a specific physical-layer packet representation.

\fakepar{Deduplication Logic}
The central design problem in FeBEx is deciding whether an arriving packet is the first observed copy of a logical uplink or a redundant duplicate that should be suppressed. To do this within bounded switch memory, FeBEx uses a two-level hash design. First, an \textit{index hash} maps the packet identifier $(\texttt{dev\_addr}, \texttt{fcnt})$ to a register slot. Second, a \textit{verification hash} is stored in that slot to reduce the chance of treating two different packets as the same entry. Each slot additionally stores an epoch value, allowing stale entries to expire without explicit deletion.

Operationally, for each arriving packet, the switch computes an index and verification key from $(\texttt{dev\_addr}, \texttt{fcnt})$, reads the stored tuple at the selected register slot, and compares the stored key and epoch against the current values, as illustrated in Fig.~\ref{fig:dedupalgo}. If both match, the packet is treated as a duplicate and suppressed. Otherwise, the switch updates the slot with the new key and current epoch, and the packet is forwarded as the first valid copy. This design keeps the data-plane state compact while enabling duplicate detection at line rate.

\fakepar{Epoch-Based Expiry}
A deduplication entry should not persist indefinitely, since old packets must not suppress future packets that happen to reuse the same identifier space after sufficient time has passed. FeBEx addresses this using epochs. The control plane periodically increments a global epoch counter, and each register entry stores the epoch in which it was last written. On lookup, if the stored epoch differs from the current epoch, the entry is treated as stale and can be safely overwritten. This avoids the need for explicit per-entry deletion and provides a simple mechanism for bounding the lifetime of deduplication state. In practice, epoch intervals in the range of a few seconds provide a good tradeoff between state freshness and duplicate suppression.

\fakepar{Tenant-Aware Forwarding}
Because FeBEx is intended for federated and potentially multi-tenant deployments, duplicate suppression alone is insufficient; packets must also be routed to the correct backend. FeBEx performs tenant steering using Longest Prefix Match (LPM) over \texttt{dev\_addr}. The control plane assigns each tenant a contiguous address prefix and installs the corresponding forwarding entries into the switch. On ingress, the switch matches the packet's \texttt{dev\_addr} against these prefixes, selects the egress port for the appropriate LNS backend, and rewrites the destination Ethernet address before forwarding. This design is simple, scalable, and avoids maintaining per-device forwarding state in the switch.

\fakepar{Attribution and Receipt Cloning}
A key requirement in Helium-style deployments is that duplicate suppression must not destroy attribution information. Although only one packet copy should continue toward the backend, the system must still preserve information about which gateway legitimately delivered the first valid copy. FeBEx supports this through ingress-to-egress cloning. When a packet is identified as the first valid copy of a logical uplink, the switch not only forwards it to the selected tenant backend but also clones it to a cloud endpoint. The cloned packet retains the gateway identifier, enabling downstream Proof-of-Coverage-related accounting and auditing. In our implementation, this is realized using a P4 clone action with a controller-configured clone session.

\fakepar{Control Plane Responsibilities}
The FeBEx control plane is intentionally lightweight. Its role is not to process packets directly, but to provide the data plane with the configuration needed for correct operation. Specifically, the controller installs LPM entries for tenant steering, configures the clone session used for receipt mirroring, initializes and updates deduplication-related state, and periodically rotates the epoch value used for entry expiry. This separation keeps the per-packet logic inside the switch fast and simple, while allowing reconfiguration and policy updates at runtime.

\fakepar{Design for Controlled Comparison}
To measure the effect of in-network deduplication, FeBEx also supports a routing-only baseline mode in which duplicate suppression is disabled and all packets are forwarded. This mode is controlled through a small register-based toggle in the P4 program. When disabled, the switch still performs tenant-aware routing and cloning, but no packets are suppressed. This allows direct A/B comparison between conventional forwarding and FeBEx-style in-network deduplication under the same topology and traffic conditions.
