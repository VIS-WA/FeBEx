\section{Implementation}
\label{sec:implementation}

This section summarizes the FeBEx implementation at a high level, focusing on how the design is realized in the prototype rather than on code-level details. FeBEx is implemented using three main components. First, the \textit{P4 data plane}, running on BMv2, parses the FeBEx metadata header and performs tenant steering, duplicate suppression, and receipt cloning. Second, a \textit{P4Runtime control plane}, implemented in Python using Finsy, installs tenant forwarding entries, configures clone sessions, and rotates the epoch value used for state expiry. Third, an experiment harness, built using Mininet and Scapy, generates traffic, captures outputs, and computes the metrics used in evaluation.

\fakepar{Packet and Traffic Path}
Each gateway sends UDP packets carrying FeBEx metadata of the form $(\texttt{dev\_addr}, \texttt{fcnt}, \texttt{gw\_id})$. On arrival at the switch, the packet is parsed, mapped to a tenant backend using \texttt{dev\_addr}, checked against deduplication state, and either forwarded or suppressed. For each unique logical uplink, FeBEx forwards exactly one copy to the correct tenant backend and mirrors that first-forwarded copy to the cloud path for attribution. Subsequent duplicates of the same logical uplink are dropped in the switch.

\fakepar{Switch Pipeline Flow}
The switch ingress pipeline follows a simple sequence of operations, as illustrated in Fig.~\ref{fig:impl-switch-flow}. The parser first extracts Ethernet, IPv4, UDP, and FeBEx metadata fields. The ingress pipeline then performs tenant steering using an LPM table over \texttt{dev\_addr}, producing the tenant-specific egress decision. Next, the switch executes the deduplication check by computing a hash index and verification key, reading the corresponding register state, and comparing the stored values against the current packet and epoch. If the packet is identified as a duplicate, it is dropped. Otherwise, the corresponding state is updated, the packet is forwarded to the tenant backend, and a clone is generated for receipt handling.

\fakepar{Controller Operation}
The controller is responsible for configuring the switch but does not participate in per-packet processing. At startup, it connects to the switch over P4Runtime, loads the compiled pipeline, installs tenant-specific LPM forwarding entries, initializes deduplication-related state, and configures the clone session used for receipt mirroring. During runtime, it periodically updates the global epoch register according to the chosen expiry interval. This design keeps the switch fast and self-contained while allowing runtime control over policy and state lifetime.

\fakepar{Deduplication State Layout}
FeBEx uses a compact register-based state design. The register array \texttt{dedup\_keys[index]} stores a 32-bit verification key for each occupied slot, while \texttt{dedup\_epochs[index]} stores the corresponding 16-bit epoch value. A separate singleton register, \texttt{current\_epoch[0]}, holds the epoch value currently enforced by the controller. Another singleton register, \texttt{dedup\_enabled[0]}, acts as a runtime toggle for enabling or disabling duplicate suppression during A/B comparisons. This compact layout avoids expensive per-device state while supporting simple and efficient lookup-and-update operations in the data plane.

\fakepar{Switch Variants for Boundary-Leakage Mitigation}
To study the effect of epoch-boundary behavior and collision tradeoffs, we implement and compare three switch variants. \textit{V1} is the baseline design, which treats a packet as a duplicate only if the stored epoch exactly matches the current epoch; this makes it vulnerable to boundary leakage when repeated packets straddle an epoch transition. \textit{V2} extends the design with a sliding-window check over both the current and previous epoch, allowing cross-boundary duplicates to still be recognized and suppressed. \textit{V3} adopts a dual-register Bloom-style design to strengthen collision behavior and reduce false positives, though its boundary behavior remains similar to V1 unless combined with additional temporal logic.

\fakepar{Implementation Choices}
Several implementation choices keep the prototype simple while preserving the essential system behavior. FeBEx uses a two-hash approach, with one hash selecting the register index and the other acting as a verification key to reduce collision ambiguity. State expiry is implemented through epoch rotation rather than explicit deletion, which keeps the control plane lightweight. Tenant isolation is achieved through prefix-based LPM forwarding, avoiding per-device forwarding state. Finally, the switch performs clone-on-first-forward so that attribution is preserved even when redundant copies are suppressed.
